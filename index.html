Stack-Light Sequencer

01Overview

An I/O tree built from scratch.

This project is a graduate lab assignment (Lab 8) in the EE 5340 Advanced PLC course. The objective was to configure a full Ethernet/IP distributed-I/O subnet in Studio 5000 Logix Designer and program a ControlLogix processor to drive a stack light through a timed, operator-controlled lamp sequence using only remote Point I/O modules — no local discrete I/O on the controller chassis.

The deliverable exercises the end-to-end workflow an entry-level controls engineer performs on a real cell: building an I/O tree from scratch, assigning IP addresses on the plant subnet, setting electronic keying and the requested packet interval (RPI) on the remote adapter, aliasing raw module tags into meaningful process names, and implementing a latch/unlatch state machine with TON-timer dwell that survives pause/resume from the operator pushbuttons.

02System Architecture

One processor, one remote drop, six rungs.

The cell is organized around a single ControlLogix processor that scans a remote Point I/O chassis over Ethernet/IP. All process I/O — both the Start/Stop pushbuttons and the four stack-light lamps — is resolved through the remote adapter, which keeps the chassis wiring clean and lets the same processor later manage several distributed Point I/O drops if the cell is extended. Logically the application is a flat, linear sequencer.

Hardware Stack

1756 backplane bridged to a 1734 remote chassis

An Allen-Bradley 1756-L71 ControlLogix 5570 controller sits in slot 5 of a 1756-A10 10-slot backplane, paired with a 1756-EN2TR two-port Ethernet/IP bridge in slot 7 acting as the scanner — addressed at 131.151.52.192 on the plant ECE Department subnet.

Downstream is a 1734-AENT/B Point I/O Ethernet adapter at 131.151.52.150, configured as a 3-slot chassis (the adapter counts as the first slot) with a 20 ms RPI and compatible-module electronic keying. Slot 1 holds a 1734-IB8/C sinking input module; slot 2 holds a 1734-OB8/C relay output module wired to the four stack-light lenses.

Software Stack

Ladder under a single MainRoutine

Written entirely in Ladder Diagram in Studio 5000, under the default MainTask / MainProgram / MainRoutine. Operator-facing names (Start_PB, Blue_LA, …) are Alias tags pointing at raw adapter bits, so the ladder reads as plain English and survives any future slot renumbering.

03The Sequence

Blue, green, yellow, red — then loop.

Pressing Start fires Blue for one second, advances to Green for one second, then Yellow for one second, then Red for five seconds — long enough to let the self-flashing red lens play — and repeats. Pressing Stop extinguishes the active lamp and holds the sequence; pressing Start again resumes from the beginning of the next sequence step. Each step is a latched step bit driving its lamp alias and a TON dwell timer, with the timer-done bit firing an OTU/OTL pair to hand off to the next step.

04Ladder Rungs

Seal-in latch, first-scan init, four step rungs.

The MainRoutine hosts six rungs: a Run-latch rung that captures operator intent, a first-scan reset rung that guarantees the sequence always restarts at Step_1, and four identical sequencer rungs — one per lamp — that each energize an output, start a timer, and advance the step bit on timer-done via an OTU/OTL pair. Adding another step is a single rung insertion.

0Rung
Run seal-in latch
A series contact on Start_PB with a parallel branch on Run drives Run through a normally closed Stop_PB — pressing Start latches Run, pressing Stop breaks the seal. This single bit gates every step and output.
1Rung
First-scan sequence initialization
If Run is true and none of Step_1..Step_4 is set, latch Step_1 — guarantees the first lamp fires immediately on start and that every restart picks up from the beginning.
2Rung
Step 1 — Blue lamp
Step_1 AND Run energizes Blue_LA and starts Blue_Tmr (preset 1000 ms). On Blue_Tmr.DN: OTU Step_1, OTL Step_2.
3Rung
Step 2 — Green lamp
Step_2 AND Run energizes Green_LA and starts Green_Tmr (preset 1000 ms). On done: OTU Step_2, OTL Step_3.
4Rung
Step 3 — Yellow lamp
Step_3 AND Run energizes Yellow_LA and starts Yellow_Tmr (preset 1000 ms). On done: OTU Step_3, OTL Step_4.
5Rung
Step 4 — Red lamp & wrap
Step_4 AND Run energizes Red_LA and starts Red_Tmr (preset 5000 ms, letting the self-flashing red lens play). On done: OTU Step_4, OTL Step_1 — wrapping the sequence so it cycles indefinitely.
05Phase Progression

Tree, logic, commissioning.

01
Network & I/O Tree

Build the Ethernet/IP tree from the ground up

1756-EN2TR scanner added to the I/O Configuration, then the 1734-AENT adapter attached beneath its Ethernet port at .150, revision 5.1, chassis size 3. IB8/C and OB8/C added with fixed slots, series C, rev 3. Compatible-module keying chosen over exact-match to tolerate bench swaps; 20 ms RPI set for one-second sequence resolution without excess traffic.

02
Ladder Logic

Author the six-rung sequencer

With the I/O tree verified by toggling raw output bits from the tag browser, the logic was authored: seal-in Run latch, first-scan Step_1 init, and four OTL/OTU step rungs driven by TON timers. Raw adapter bits aliased to Start_PB / Blue_LA / … so the ladder reads in plain English.

03
Commissioning

Verify raw I/O, then the full sequence

Download, confirm all four modules report Standby/Running, then exercise each input and output in Run. Pushbuttons verified by watching the raw input bits toggle; lamps verified by forcing each alias bit. Full sequence then tested end-to-end with pause/resume — module-properties and tag-listing reports generated as the submitted deliverable.

07Outcome

Initializes clean, runs the loop, pauses without sticking.

The final deliverable is a working Point I/O stack-light sequencer that initializes cleanly on first scan, responds immediately to the Start pushbutton with a Blue-Green-Yellow-Red sequence on the configured timing, pauses cleanly on Stop without sticking a lamp, and resumes on the next Start press. Beyond the working demo, the most valuable outcome was the full round-trip experience of configuring an Ethernet/IP I/O tree from scratch and reasoning about the difference between raw module tags and descriptive alias tags — a small detail on a small lab, but the same pattern every production Logix project relies on to stay maintainable as the I/O layout evolves.
